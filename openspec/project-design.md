# GachaPlanner 项目总体设计

## 1. 产品定位

GachaPlanner（抽卡规划师）是一款面向抽卡游戏玩家的小程序工具，核心解决玩家在抽卡决策中的信息不透明问题：

- **概率可视化**：将隐藏的概率机制转化为直观的数据
- **资源规划**：帮助玩家合理分配有限的抽卡资源
- **历史追踪**：记录抽卡历史，分析个人运气趋势
- **跨游戏支持**：一个工具管理多款游戏的抽卡规划

## 3. 核心功能模块

### 3.1 概率计算引擎（core/calculator/）

**职责**：根据游戏配置计算抽卡概率分布

**设计要点**：
- 策略模式：每种保底机制对应一个计算器类
- 工厂模式：`CalculatorFactory` 根据配置自动选择计算器
- 基类约束：所有计算器继承 `BaseCalculator`，确保接口一致

```
┌─────────────────────────────────────────────┐
│           CalculatorFactory                  │
│  createCalculator(config) ──▶ Calculator     │
└─────────────────────────────────────────────┘
                      │
        ┌─────────────┼─────────────┬─────────────┐
        ▼             ▼             ▼             ▼
   ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐
   │ HardPity│  │SoftPity │  │ NoPity  │  │Guarantee│
   │   90抽  │  │ 74抽起  │  │ 纯概率  │  │ 大小保底│
   └─────────┘  └─────────┘  └─────────┘  └─────────┘
        │             │             │             │
        ▼             ▼             ▼             ▼
   ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐
   │  Luck   │  │MultiPool│  │Dynamic  │  │Combined │
   │  361值  │  │ 多池联动│  │动态概率  │  │综合计算  │
   └─────────┘  └─────────┘  └─────────┘  └─────────┘
```

### 3.2 游戏配置系统（config/）

**职责**：管理所有游戏的配置参数，支持自定义和扩展

**设计要点**：
- 双层配置：基础配置 + 用户自定义覆盖
- 模板系统：预设 5 种通用模板（mihoyo, arknights, fgo, hok, onmyoji）
- 验证机制：配置加载时自动验证参数合法性
- 版本管理：配置包含版本号，支持向后兼容

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│ 基础配置     │     │ 用户自定义   │     │  合并配置   │
│ (games/)    │  +  │ (storage)   │  ──▶│ (runtime)  │
└─────────────┘     └─────────────┘     └─────────────┘
```

### 3.3 数据持久化层（services/storage-service.js）

**职责**：管理所有用户数据的持久化、版本迁移和缓存

**设计要点**：
- 智能缓存：游戏数据常驻内存，避免重复读取存储
- 版本迁移：数据版本升级时自动迁移旧格式
- Result 模式：所有操作返回 { success, data, error }

### 3.4 规划管理系统（services/planning-storage.js）

**职责**：管理用户的卡池规划数据

**设计要点**：
- 支持多游戏、多卡池的规划
- 规划数据与游戏资源数据联动
- 支持规划的增删改查和排序

### 3.5 历史记录系统（services/history-service.js）

**职责**：记录和分析用户的抽卡历史

**设计要点**：
- 记录每次抽卡的时间、游戏、卡池、结果
- 支持历史趋势分析
- 计算个人准确度统计

### 3.6 提醒服务（services/reminder-service.js）

**职责**：管理卡池开启/截止提醒

**设计要点**：
- 本地提醒通知
- 支持一次性提醒和周期性提醒
- 与规划系统联动

## 5. 数据流设计

```
用户操作 ──▶ 页面/Page ──▶ 服务/Service ──▶ 存储/Storage
                              │
                              ▼
                         计算引擎/Core
                              │
                              ▼
                         组件展示/Component
```

**数据流原则**：
1. 单向数据流：页面发起操作，服务处理逻辑，组件展示结果
2. 状态管理：每个页面独立管理自己的 data，通过 service 共享数据
3. 事件通信：父子组件通过 properties/events，跨组件通过全局数据或 service

## 7. 关键算法

### 7.1 概率分布计算
```
累积概率 P(n) = 1 - ∏(1 - p_i)  (i=1 to n)
其中 p_i 为第 i 抽的单抽概率
```

### 7.2 软保底概率
```
p(n) = baseRate + (n - softPityStart + 1) * increment   (n >= softPityStart)
p(n) = baseRate                                         (n < softPityStart)
```

### 7.3 资源转换
```
总抽数 = primaryResource / conversionRate + secondaryResource
```

## 8. 安全与验证

### 8.1 配置验证
- 必填字段检查
- 数值范围验证（如 baseRate 在 0.001-1.0 之间）
- 逻辑一致性验证（如 softPityStart < hardPity）

### 8.2 数据安全
- 配置导出包含 checksum，防止篡改
- 用户数据存储在本地，不上传服务器
- 导入配置时验证来源合法性

## 9. 性能设计

### 9.1 缓存策略
- 游戏配置：启动时加载，全局缓存
- 用户数据：按需加载，修改时更新缓存
- 计算结果：适当缓存，避免重复计算

### 9.2 渲染优化
- 图表组件使用节流（throttle）控制重绘频率
- 大数据列表使用虚拟滚动
- 图片懒加载

## 10. 扩展性设计

### 10.1 添加新游戏
1. 在 `config/games/` 创建配置文件
2. 使用已有模板或完全自定义
3. 验证配置合法性
4. 无需修改核心代码

### 10.2 添加新保底机制
1. 在 `core/calculator/` 创建新计算器类
2. 继承 `BaseCalculator` 并实现接口
3. 在 `CalculatorFactory` 注册
4. 在 `config-validator.js` 添加验证规则

### 10.3 添加新页面
1. 在 `pages/` 创建页面目录
2. 在 `app.json` 注册页面路径
3. 如需 Tab 栏入口，在 `app.json` tabBar 配置
