## Context

`overview-redesign` 已交付多游戏资源统筹的核心功能（目标管理、资源模拟器、策略建议、冲突检测）。本次 change 为上一轮交付后的 UX 打磨，聚焦信息架构合理化、交互流程顺畅化、核心操作显性化。

当前痛点：
- **规划页**：目标设定位于页面最底部，玩家需要滑动 5-6 屏才能完成「设定目标→输入资源→查看概率」的核心流程；卡池时间在「卡池时间设置」和「设定目标」两处重复编辑
- **总览页**：目标卡片横向滚动不利于 2-3 款游戏的快速对比；策略区域存在标题重复渲染；模拟器 ±1 抽步进无实际意义

## Goals / Non-Goals

**Goals:**
- 规划页信息架构符合玩家「先定目标→再管资源→最后看概率」的认知模型
- 消除卡池时间重复设置，统一在目标表单中维护
- 总览页支持竖向快速扫视多游戏状态
- 核心操作按钮（保存目标、采用策略）视觉层级清晰、文案明确

**Non-Goals:**
- 不改动核心概率计算逻辑
- 不引入新的数据模型或存储结构
- 不修改游戏配置加载机制

## Decisions

### 1. 规划页区块顺序：目标 → 保底 → 资源 → 概率 → 图表 → 攒抽预测 → 版本信息
- **Rationale**：二游玩家的核心任务是「我要抽谁」「我有多少资源」「能不能抽到」。目标设定是起点，应放在最上方；版本信息是最低优先级的元数据，沉底
- **Alternative considered**：将概率展示和目标设定并排——但手机屏幕宽度有限，不适合并排布局

### 2. 卡池时间只保留在目标表单中，pool-settings 改为只读联动
- **Rationale**：同一目标的卡池时间理应是唯一的。用户在目标中设定后，攒抽预测区域自动读取并显示，无需二次编辑
- **Implementation**：`onTargetPoolStartDateChange` / `onTargetPoolEndDateChange` 中同步更新 `poolStartDate` / `poolEndDate`；pool-settings 区域移除 date picker，改为只读文本展示

### 3. 目标卡片从横向滚动改为竖向列表
- **Rationale**：二游玩家通常同时管理 2-3 个游戏，竖向列表可以一眼看到全部状态，横向滚动需要滑动才能对比
- **Trade-off**：竖向列表占用更多垂直空间，但首屏信息密度更高，玩家无需交互即可获取全部信息

### 4. 模拟器步进改为 ±10, ±50
- **Rationale**：抽卡是 160 原石=1 抽的离散单位，±1 抽的调整对规划没有实际价值。±10 和 ±50 更符合玩家的操作习惯（整十整百调整）
- **Note**：保留点击数值直接输入的快捷方式，满足精确调整需求

## Risks / Trade-offs

| Risk | Mitigation |
|------|-----------|
| 已习惯旧布局的用户需要适应新顺序 | 新顺序符合直觉，适应成本低；可在更新日志中说明 |
| 竖向列表在目标多时导致页面过长 | 目标数量通常 ≤3（同时玩的游戏数），不会过长 |
| 移除 pool-settings 中的日期编辑后，用户找不到修改入口 | 在 pool-settings 区域显示「卡池时间已在目标中设定」提示，并支持点击跳转到目标表单 |
