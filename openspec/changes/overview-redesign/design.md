## Context

### 当前状态

`pages/overview/overview` 当前是一个展示型页面，包含：
- `summary-card`: 展示总抽数、游戏数量、平均概率、综合概率
- `games-section`: 各游戏达成概率列表，点击跳转规划页
- `strategy-advice`: 一句文本建议（如"建议集中投入崩铁"）

页面数据来源：`gameService.getGameResources()` + `CalculatorFactory` 逐游戏计算概率。综合概率由 `CombinedProbabilityCalculator` 计算，假设所有资源集中到最优游戏。

现有组件：`overview-card`、`game-card`（在 `components/` 中）。

### 约束

- **微信小程序平台限制**：无 WebGL、canvas 2d 有限、本地存储 10MB
- **手势冲突**：页面滚动与滑块拖拽天然冲突，小程序中无完美解决方案
- **交互习惯**：小程序用户不熟悉「长按」操作；官方规范推荐点击、滑动、轻触
- **信息密度**：iPhone 14 一屏约 844pt，超过一屏的内容需要滚动才能看到
- **数据纯本地**：`wx.getStorageSync`，无后端
- **主题系统已固化**：莫兰迪暖色系（`#C4A77D` 主色），组件需延续此风格
- **概率计算引擎不可改动**：`CalculatorFactory`、`CombinedProbabilityCalculator`、`StrategyOptimizer`、`SuggestionGenerator` 均复用现有实现
- **独立开发资源有限**：优先复用，新增组件控制在 4 个以内

## Goals / Non-Goals

**Goals:**
- 用户打开总览页后**首屏内**（不滚动）能感知「我的目标够不够」
- 用户可以在总览页直接调整资源分配并实时看到概率变化（≤300ms 反馈）
- 系统自动生成 2-3 个可执行的策略方案，用户可预览效果后确认应用
- 当多游戏目标存在资源冲突时，系统自动检测并提示
- 保持现有莫兰迪暖色系视觉风格，新组件风格一致
- 兼容旧数据：未设定目标的游戏按现有逻辑展示资源概况

**Non-Goals:**
- 不改动单游戏规划页（`pages/planning/planning`），只增加目标数据写入
- 不做卡池排期/时间线规划（Phase 2）
- 不做充值金额换算（Phase 2）
- 不做社区分享/排行榜
- 不做云端同步/跨设备
- 不改动核心概率计算引擎
- 不使用滑块（slider）作为资源调整的主要交互方式

## Decisions

### Decision 1: 目标数据存储在 PlanningStorage 而非独立存储

**选择**: 扩展 `PlanningStorage` 的 `targets` 字段，而非新建存储服务。

**rationale**: PlanningStorage 已经管理每个游戏的规划数据，目标属于规划的一部分。复用现有存储键前缀 `gacha_planning_` 和版本迁移机制，减少维护成本。

**Alternative considered**: 新建 `GoalStorage` 独立服务。Rejected：增加复杂度，且目标与资源数据天然耦合（概率计算需要同时读取）。

### Decision 2: 资源分配模拟使用「总资源守恒」模型

**选择**: 用户在模拟器中调整某游戏资源时，总抽数保持不变，增加量自动从其他游戏扣除。

**rationale**: 真实场景下用户的总资源是固定的（只是分配在不同游戏），守恒模型最贴近现实。同时避免了「凭空变出资源」的困惑。

**扣除优先级策略**: 优先从「达成概率最高的游戏」扣除，保证整体损失最小。例如用户给崩铁加 30 抽，从原神扣（原神概率 75% → 仍有 45%），而非从已经很低的游戏扣。

**边界情况处理**:
- 多游戏概率相同时：从**当前抽数最多**的游戏扣除（资源充裕的游戏更经得起削减）
- 单游戏用户：守恒逻辑不触发（只有一个游戏，资源无法转移），隐藏模拟器调整按钮
- 某游戏被扣至 `minPulls`（默认 0）时：顺延至下一个概率最高的游戏继续扣除

**Alternative considered**: 允许总资源变化（模拟「假如我多攒了 X 抽」）。Rejected：独立开发阶段保持简单，此功能可在 Phase 2 作为「预算模拟」扩展。

### Decision 3: 策略方案由前端实时生成，不预计算存储

**选择**: 每次页面加载时，基于当前资源分配实时生成策略方案。

**rationale**: 策略方案完全由当前资源数据和目标决定，没有需要持久化的状态。实时生成保证数据一致性，且计算量小（最多 3 个游戏 × 3 种方案的排列组合）。

**生成算法**:
```
方案A（现状）: 保持当前资源分配
方案B（集中）: 将所有资源集中到达成概率提升最大的单一游戏
方案C（均衡）: 当总资源足够时，均分使各游戏概率 ≥60%
方案D（补充预算）: 当总资源不足使所有游戏达成概率 ≥80% 时，计算所需额外抽数（向上取整到 10 的倍数），展示「补充 X 抽后的效果」
```

**动态生成规则**:
- 单游戏用户：只生成方案 A、B（方案 C/D 无意义）
- 多游戏 + 总资源充足：生成方案 A、B、C
- 多游戏 + 总资源不足：生成方案 A、B、D（用「补充预算」替代不可行的「均衡」）

**Alternative considered**: 服务端预计算（无服务端，不可行）。Rejected by constraint。

### Decision 4: 目标卡片状态色按概率阈值分三档

**选择**: ≥80% 绿色 / 50%~80% 黄色 / <50% 红色。

**rationale**: 三档足够传达「稳/可搏/危险」的决策信息，过多档位增加认知负担。80% 是抽卡玩家普遍接受的「稳拿」心理阈值。

**Alternative considered**: 连续色温标（0-100% 渐变色）。Rejected：小程序中连续色温视觉区分度低，且玩家不需要知道「67% 和 72% 的颜色差别」。

### Decision 5: 模拟器交互使用按钮步进 + 点击输入，不使用滑块

**选择**: 提供 `[−10] [−1] [+1] [+10]` 按钮（步进调整）+ 中间数字区域点击输入（弹出数字键盘）。下方保留**静态**进度条（只展示，不可交互）。

**rationale**:
- 微信小程序的 `slider` 组件与页面滚动存在天然手势冲突：用户拖拽滑块时极易触发页面滚动，反之亦然
- 按钮步进提供确定性的调整（每次 ±1 或 ±10），用户心理预期明确
- 点击数字输入适合大额调整，比滑块拖拽更精准
- 静态进度条展示「占总资源比例」，满足视觉化需求但不引入交互冲突

**Alternative considered**: 纯滑块拖拽。Rejected：小程序平台手势冲突问题无法完美解决，用户体验差。
**Alternative considered**: 数字输入框直接输入。Rejected：需要唤起键盘、占用屏幕、打断流程；作为辅助方式存在，不作为主要交互。

### Decision 6: 模拟器区域默认折叠，首屏只展示概要

**选择**: 资源分配模拟器默认处于折叠态，只展示各游戏的当前分配概况（进度条 + 概率）。点击「展开」后才显示调整按钮。

**rationale**:
- 首屏信息密度过高会导致用户打开页面后需要滚动才能看到核心内容
- 折叠态下模拟器区域高度约 200rpx，加上目标卡片和冲突提示后，首屏总高度控制在 750rpx 以内（适配 iPhone 14 及同级别设备）
- 当检测到资源冲突时，点击冲突提示的「帮我算一下」按钮可自动展开模拟器

**折叠态展示**:
```
📊 资源分配模拟器              [展开 ▼]
原神 ████████████░░ 100抽  75%
崩铁 ██████░░░░░░░░  80抽  45%
```

### Decision 7: 目标卡片删除操作使用右上角「···」菜单，不使用长按

**选择**: 每个 `goal-card` 右上角放置 `···` 按钮，点击后弹出 `wx.showActionSheet`（编辑 / 删除 / 取消）。

**rationale**:
- 微信小程序用户几乎没有「长按」操作习惯，该功能不可发现
- 长按 500ms 的反馈延迟让用户不确定操作是否生效
- 微信官方 UI 规范中，列表项的操作入口通常使用右上角菜单或左滑
- 小程序没有原生 swipe-cell 组件，左滑需手写 touch 事件，开发成本高

**Alternative considered**: 长按删除。Rejected：不可发现、反馈延迟、不符合平台习惯。
**Alternative considered**: 左滑删除。Rejected：无原生支持，自定义实现成本高，且与页面滚动冲突。

### Decision 8: 冲突提示使用「帮我算一下」主按钮 + 「暂不处理」文字链接

**选择**: 冲突提示卡片的主按钮文案为「帮我算一下怎么分配」，次要操作为弱化的文字链接「暂不处理」。去掉「我知道了」按钮。

**rationale**:
- 「我知道了」暗示用户看完就完事了，但冲突实际上没有解决，下次打开页面又会弹出，形成骚扰
- 「帮我算一下」直接引导用户进入行动（展开模拟器），缩短决策路径
- 「暂不处理」作为弱化链接，让不想处理的用户安静地 dismiss，不打断体验

### Decision 9: 策略卡片增加「预览效果」中间态

**选择**: 每个策略卡片提供「预览效果」按钮，点击后临时将该方案的资源分配应用到上方的模拟器区域（不保存），用户满意后再点击「确认采用此方案」。

**rationale**:
- 用户直接「应用」方案的心理压力大（担心改错、不好恢复）
- 「预览」让用户先看到效果再决策，降低决策成本
- 预览态下模拟器区域高亮提示「当前为预览，点击确认采用或取消」
- 取消预览后自动恢复原始分配

### Decision 10: 目标卡片强化概率数字作为视觉焦点

**选择**: 目标卡片中，概率数字使用 `font-size: 48rpx` + `font-weight: bold`，作为卡片的视觉焦点。「还差 X 抽」使用功能色（如 `#C47070` 红色）突出。「目标名称」使用 `font-size: 28rpx` 作为二级信息。

**rationale**:
- 用户看目标卡片的第一诉求是「稳不稳」，概率是最核心的信息
- 当前设计中概率数字与其他信息同层级，视觉焦点不明确
- 大号概率数字配合背景色（绿/黄/红），用户扫一眼即可判断状态

### Decision 11: 冲突检测阈值固定为 80%

**选择**: 当 ≥2 个游戏的目标达成概率 < 80% 时触发冲突提示。

**rationale**: 80% 是玩家心理安全线。两个游戏都低于此线意味着「两个都想要但两个都不稳」，正是需要统筹决策的典型场景。

**Alternative considered**: 可配置阈值。Rejected：独立开发阶段保持简单，阈值可后期开放为高级设置。

### Decision 12: 重置按钮放在模拟器区域底部，并提供撤销机制

**选择**: 「重置」按钮位于模拟器区域底部（用户操作完后才看得到，降低误触概率）。点击重置后显示 toast「已重置为原始分配」，并提供「撤销」按钮（3 秒内有效）。

**rationale**:
- 重置按钮放在标题栏右侧时，用户可能在浏览过程中误触，导致调整成果丢失
- 底部位置符合操作流（从上到下浏览，到底部后决定重置）
- 撤销机制给用户一次「反悔」机会，减少误操作焦虑

## Risks / Trade-offs

| Risk | Severity | Mitigation |
|------|----------|------------|
| 概率计算频繁触发导致卡顿 | 中 | 模拟器调整使用 300ms throttle；仅计算受影响的游戏；进度条动画用 CSS transition 而非 JS |
| 目标数据模型后续扩展困难 | 低 | targets 数组结构预留 `type`、`rarity`、`constellations` 等扩展字段；使用版本号机制（DATA_VERSION）支持迁移 |
| 多游戏用户稀少，功能使用率不高 | 中 | 单游戏用户看到「添加目标」引导后，仍可将其作为单游戏目标管理工具使用；空状态设计为「设定目标后体验更佳」而非强制 |
| 资源守恒扣除逻辑引发困惑 | 低 | UI 明确提示「从原神挪 30 抽到崩铁」；提供「重置」按钮 + 撤销机制一键恢复 |
| 策略方案推荐不准确 | 低 | 方案文案使用保守表述（如「建议」而非「必须」）；展示各方案的具体概率数字让用户自行判断；不自动应用任何方案，需用户手动确认 |
| 首屏信息仍可能超出小屏设备 | 低 | 目标卡片支持横向滑动，减少垂直占用；冲突提示条件展示（仅冲突时出现）；模拟器默认折叠 |

## Migration Plan

1. **数据兼容**: PlanningStorage 读取旧数据（无 targets 字段）时，targets 默认为空数组，页面降级展示现有资源概况
2. **版本升级**: 首次写入目标时，数据版本从 `1.0.0` 升级到 `2.0.0`
3. **回滚**: 删除目标即可恢复为现有展示逻辑，无需数据回滚
4. **组件发布**: 新组件（goal-card、allocation-bar、strategy-card、conflict-alert）独立开发，不影响现有页面；overview 页面改造完成后一次性替换

## Open Questions

### Q1: 目标设定的入口位置

**决策**: 点击「+ 目标」→ 弹出**游戏选择器**（轻量，仅选择游戏）→ 选择后**跳转到对应游戏的 planning 页**并自动展开目标设定区域。

**Rationale**:
- 目标设定涉及字段较多（目标名称、类型、期望命座/精炼、卡池时间），在当前页弹出复杂表单会严重挤占首屏空间
- planning 页已有完整的游戏资源上下文（当前抽数、保底进度等），用户设定目标时需要参考这些数据
- 小程序用户习惯「选择一个东西 → 去专门页面配置」的流程（类似微信设置中的层级导航）
- 游戏选择器用 `wx.showActionSheet` 即可实现，轻量无跳转负担；选择后跳转到 planning 页携带 `gameId` 参数和 `autoFocusTarget=true` 标记

### Q2: 策略方案数量

**决策**: **动态生成**，规则如下：

| 用户场景 | 生成方案 | 说明 |
|----------|----------|------|
| 单游戏用户 | 现状、集中 | 只有两种有意义 |
| 多游戏 + 总资源充足（能使各游戏≥60%）| 现状、集中、均衡 | 三种全展示 |
| 多游戏 + 总资源不足 | 现状、集中、补充预算 | 「均衡」不可行，替换为预算建议 |

- 「均衡」方案仅在总资源足够时生成，避免展示无意义的方案
- 「补充预算」方案计算使所有游戏达成概率≥80% 所需的额外抽数，向上取整到 10 的倍数
- 最多 3 个，最少 2 个，保证卡片区域不空且不冗长

### Q3: 是否保留现有 overview-card 组件

**决策**: **复用并降级为模拟器内的 summary 行**，而非独立卡片。

**Rationale**:
- redesign 后首屏核心信息是「目标卡片 + 冲突提示 + 折叠模拟器」，旧 overview-card 的「总抽数/平均概率/综合概率」作为独立卡片价值降低且占用首屏空间
- 但用户仍需要快速感知「我总共多少抽」，所以将统计信息压缩为模拟器折叠态内的一行 summary：`总资源: 180抽 | 目标: 2个 | 平均达成率: 60%`
- 对于**无目标用户**（降级模式），保留原有 overview-card 完整展示，作为 fallback
- 复用现有样式结构，减少重复开发

```
┌──────────────────────────────────────────────────────────────┐
│                    pages/overview/overview                    │

│                            ▼                                  │
│                    ┌───────────────┐                          │
│                    │ conflict-alert│◄── conflicts[]           │
│                    │               │───► taphelp → 展开模拟器   │
│                    └───────┬───────┘                          │
│                            │                                  │

│                            ▼                                  │

│              │  allocation-bar          │                      │
│              │  apply ────► 持久化保存  │                      │
│              └─────────────────────────┘                      │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

**数据流向**:
1. `overview.js` 从 `PlanningStorage` / `GameService` 加载数据，生成 `goals[]`、`allocations[]`、`conflicts[]`
2. `goals[]` 绑定到 `goal-card` 列表
3. `conflicts[]` 绑定到 `conflict-alert`
4. `allocations[]` 绑定到 `allocation-bar` 列表
5. 用户调整 `allocation-bar` → 触发 `change` → `overview.js` 调用守恒计算 → 更新所有 `allocation-bar` 的 `currentPulls` / `probability`
6. `overview.js` 根据最新 allocations 重新生成 `strategies[]` → 绑定到 `strategy-card` 列表
7. 用户点击 `strategy-card` 的「预览」→ `overview.js` 将临时 allocations 写入 `allocation-bar`（通过数据绑定），并设置 `isPreviewing=true`
8. 用户点击「确认采用」→ `overview.js` 保存到 `PlanningStorage` → 刷新全页数据
